## はじめに — 「エンジニア何人採れば足りますか？」という問いの限界

「来期、エンジニアあと何人必要ですか？」

CTOや技術責任者であれば、経営会議で一度はこの問いを受けたことがあるはずです。そして多くの場合、この問いに対して「あと3人」「5人は欲しい」と人数で答えてきました。

しかし、この問い自体がすでに機能しなくなりつつあります。

象徴的な事例がKlarnaです。同社は2022年末に5,527人だった従業員を、2025年には約2,900人まで削減しました。その間、売上は倍増しています。CEOのSebastian Siemiatkowskiは、AIが [853人分のカスタマーサービス業務を代替した](https://www.cnbc.com/2025/05/14/klarna-ceo-says-ai-helped-company-shrink-workforce-by-40percent.html) と公言しました。

これを「AIで人を切った」と読むのは表層的です。実際に起きたのは、組織のケイパビリティ構造の転換です。同じアウトプットを、異なるリソース配置で実現した。つまり「何人いるか」ではなく「何ができるか」で組織を設計し直したということです。

本記事では、技術組織のサイジングを **「人数」から「ケイパビリティ」へ** 転換するためのフレームワークを提示します。

:::message
この記事は「人を減らせ」という主張ではありません。AIによって技術組織の投資対効果の計算式が変わりつつある中で、CTOが経営に対してどう組織を語るべきかを考えるものです。
:::

## 「人数」で語ることの3つの罠

### 罠1: ヘッドカウント ≠ アウトプット

エンジニアの生産性には大きなばらつきがあります。これは以前から知られていた事実ですが、AIツールの普及によってこの差はさらに拡大しています。

AIツールを使いこなすエンジニアと、従来の手法で開発するエンジニアでは、同じ1人でもアウトプットが数倍異なるケースが出てきました。ある支援先では、GitHub Copilotの導入後、チーム内の実装速度のばらつきが2倍から5倍に広がったという報告がありました。

人数を増やせばアウトプットが比例して増えるという前提は、もはや成り立ちません。

### 罠2: AIツールが人数計画を狂わせる

[GitHubの調査](https://github.blog/news-insights/research/research-quantifying-github-copilots-impact-on-developer-productivity-and-happiness/) によれば、GitHub Copilotの利用で開発工数が推定34%削減され、エンジニア1人あたり週6時間の節約効果があるとされています。

一方で、[METRの研究](https://metr.org/blog/2025-07-10-early-2025-ai-experienced-os-dev-study/)（2025年）では、経験豊富なOSS開発者がAIを使った場合、タスク完了時間がむしろ **19%増加** したという結果も出ています。

この矛盾が示すのは、「AIを入れたから人数を減らせる」という単純な計算が危険だということです。生産性の変化はタスクの種類、エンジニアのスキル、ツールの習熟度によって大きく異なります。

### 罠3: 採用市場のミスマッチ

経済産業省の推計では、2030年にIoT・AI分野の先端人材が **55万人不足** する一方、従来型ITサービスに従事する人材は **10万人余剰** になるとされています。

つまり「エンジニアを増やす」と言ったとき、採れるエンジニアと必要なエンジニアが一致しない構造的問題があります。人数の議論をしている限り、この質的なミスマッチは見えてきません。

技術顧問として複数の組織を支援する中で、「人が足りない」と言っている企業の多くは、実際には「人の配置が合っていない」ケースがほとんどでした。必要なのは増員ではなく、既存リソースの再配置と、AIによるレバレッジの設計だったのです。

## Two-Pizza TeamからOne-Pizza Teamへ

Amazonが提唱した「Two-Pizza Team」は、5〜8人のチームが自律的にプロダクトを持つという組織設計の定番でした。

AI時代に入り、この前提が変わりつつあります。

| 時代 | チームモデル | 典型的な人数 | 特徴 |
|------|-------------|-------------|------|
| 従来 | Two-Pizza Team | 5〜8人 | フルスタック構成、自律的 |
| 現在 | One-Pizza Team | 2〜3人 + AIツール | AIがジュニア〜ミドルの作業を代替 |
| 将来 | Solo + Agents | 1人 + AIエージェント群 | シニアエンジニアがAIチームを統率 |

[Jampa Uchoaの分析](https://www.jampa.dev/p/the-rise-of-one-pizza-engineering) が指摘するように、AIエージェントの成熟により「One-Pizza Team」で以前のTwo-Pizza Teamと同等かそれ以上のベロシティを出せるケースが増えています。Cognition LabsのDevinはわずか10人のチームで開発されましたし、Mistral AIも少人数で複数のAIモデルをリリースしています。

ただし、個人的に感じるのは、チームサイズの縮小は「人が要らなくなる」という話ではないということです。むしろ、1人1人に求められる判断力と設計力が上がる。AIが実装の速度を上げてくれる分、何を作るか・作らないかの意思決定の質が、チームの成果を直接左右するようになっています。

## フレームワーク: ケイパビリティ基準の組織サイジング

人数ではなく、組織が持つべきケイパビリティ（能力・機能）を軸にサイジングを考えるフレームワークを提示します。3つのステップで構成されます。

### ステップ1: ケイパビリティの棚卸しと分類

まず、技術組織に必要なケイパビリティを洗い出し、以下の3カテゴリに分類します。

| 分類 | 定義 | 具体例 |
|------|------|--------|
| **Human-Core** | 人間でなければ遂行できない | アーキテクチャの意思決定、ステークホルダー交渉、技術戦略の策定、組織文化の醸成 |
| **AI-Replaceable** | AIで十分に代替可能 | 定型テスト作成、ドキュメント生成、定型的なCRUD実装、コードフォーマット |
| **AI-Amplifiable** | 人間がAIと組むことで大幅に増幅される | コードレビュー、バグ調査、設計案の比較検討、技術リサーチ、リファクタリング |

:::message
重要なのは「AI-Replaceable」を即座にゼロ人にすることではなく、この分類を組織として共有し、投資の優先順位を意識することです。
:::

### ステップ2: レバレッジ比率の算出

各ケイパビリティについて、「1人あたりインパクト × AI倍率」を試算します。

```
レバレッジ比率 = 人間の基本アウトプット × AI増幅係数
```

たとえば、コードレビューという業務を考えます。

- 従来: シニアエンジニア1人が1日5PR をレビュー
- AI増幅後: 同じエンジニアがAIの一次レビューを活用して1日15PRをレビュー
- レバレッジ比率: 3.0x

このレバレッジ比率が高いケイパビリティほど、少人数で大きなインパクトを出せます。逆に、Human-Coreな領域はレバレッジ比率が1.0xに近く、ここには十分な人数を確保する必要があります。

支援先で見た事例では、この計算を実際にやってみると「レビューに3人張り付いていたが、AIレビュー導入後は1人+週次の深いレビュー会で品質が上がった」といった再配置が自然に見えてくることが多いです。

### ステップ3: ポートフォリオ配置（Hub-and-Spoke型）

ケイパビリティの分類とレバレッジ比率をもとに、組織構造を設計します。推奨するのは **Hub-and-Spoke型** です。

**Hub（中核チーム）:**
- Human-CoreとAI-Amplifiableの高レバレッジ領域を担当
- 正社員のシニアエンジニア中心
- プロダクトの意思決定権を持つ
- 目安: プロダクトあたり2〜4人

**Spoke（専門家ネットワーク）:**
- 特定領域の専門知識をオンデマンドで提供
- 業務委託・顧問・パートタイムの専門家
- セキュリティ、インフラ、データエンジニアリングなど
- 必要なときに必要な分だけ

**AIレイヤー:**
- AI-Replaceable領域の自動化
- AI-Amplifiable領域のツール基盤
- CIへの組み込み、エージェントの運用

```
         ┌─────────────┐
         │   Hub (中核)   │
         │  2〜4人/製品   │
         └──────┬──────┘
                │
    ┌───────────┼───────────┐
    │           │           │
┌───┴───┐ ┌───┴───┐ ┌───┴───┐
│Spoke A│ │Spoke B│ │Spoke C│
│セキュリティ│ │インフラ │ │データ  │
└───────┘ └───────┘ └───────┘
    ↑           ↑           ↑
    └───── AIレイヤー ─────┘
    （自動テスト・レビュー・監視）
```

## 実践: サイジング見直しの4フェーズ

フレームワークを理解しても、いきなり組織を変えることはできません。段階的に進めるための4フェーズを整理します。

### Phase 1: 現状のケイパビリティマップ作成（1〜2週間）

チームが日常的に行っている業務を洗い出し、前述の3カテゴリに分類します。

やり方はシンプルです。各エンジニアに1週間のタイムログをつけてもらい、業務を集約・分類します。ここで重要なのは、「何をしているか」だけでなく「その業務が誰にしかできないか」を問うことです。

### Phase 2: AI代替・増幅のPoC実施（1〜2ヶ月）

AI-ReplacableとAI-Amplifiableに分類された業務について、実際にAIツールを導入してPoCを行います。

ポイントは、 **全社展開ではなく1チームで小さく試す** ことです。GitHub Copilot、Claude、Cursor、自動テスト生成ツールなど、領域ごとに適切なツールを選定し、実際のレバレッジ比率を計測します。

ある支援先では、このPoCの段階で「思ったより効果が出ない領域」と「想定以上に効果がある領域」が明確になり、当初の計画を大幅に修正しました。机上の分類と実測値にはほぼ必ずギャップがあるので、このフェーズは省略しないことを強く推奨します。

### Phase 3: チーム再編と役割再定義（2〜3ヶ月）

PoCの結果をもとに、Hub-and-Spoke型への段階的な移行を行います。

| 施策 | 内容 |
|------|------|
| 役割の再定義 | 「実装者」から「設計者+AIオーケストレーター」へ |
| 評価基準の更新 | コード量ではなく、意思決定の質と影響範囲で評価 |
| スキル投資 | AIツール活用研修、プロンプトエンジニアリング |
| 配置転換 | AI-Replaceable業務の担当者を、AI-Amplifiable領域へシフト |

:::message alert
配置転換は慎重に進めてください。「AIに置き換わったから異動」という伝え方ではなく、「より高いレバレッジを出せる領域で力を発揮してほしい」というキャリア成長の文脈で語ることが重要です。
:::

### Phase 4: 継続的なキャリブレーション（四半期ごと）

AIツールの進化は速く、半年前の前提が通用しなくなることは珍しくありません。四半期ごとにケイパビリティマップとレバレッジ比率を見直し、組織のサイジングを微調整します。

ここで意識すべきは、「縮小」方向だけでなく「拡張」方向の判断もあるということです。新しいプロダクトラインの立ち上げ、AIではカバーしきれない領域の発見など、人を増やすべきタイミングも当然あります。

## 「縮小」ではなく「再配置」— 経営への説明の仕方

このフレームワークを経営会議で使うとき、最も重要なのは文脈の設定です。

「AIでエンジニアを減らせます」と言った瞬間、議論は人員削減の話になります。そうではなく、 **「同じ投資額で、組織が出せるアウトプットの上限を引き上げる」** という文脈で語るべきです。

具体的には、以下のような構造で説明します。

**従来の投資構造:**
- エンジニア10人 × 年収800万円 = 8,000万円/年
- アウトプット: 機能A, B, Cの開発・保守

**ケイパビリティ基準の投資構造:**
- Hub: シニア4人 × 年収1,000万円 = 4,000万円/年
- Spoke: 専門家3人 × 平均400万円 = 1,200万円/年
- AIツール・基盤: 800万円/年
- 合計: 6,000万円/年
- アウトプット: 機能A, B, C, D, Eの開発・保守 + 品質向上

数字は仮のものですが、論点は明確です。 **総コストを下げながらアウトプットを増やす** ことが可能になる。そしてその差分2,000万円を、新規事業への投資や、Hub人材の報酬引き上げに回せる。

[Prosum社のレポート](https://www.prosum.com/2026/01/14/5-ways-ai-augmented-developers-will-change-it-teams-in-2026/) でも指摘されているように、2026年のIT組織はチームが縮小する一方で **シニアリーダーシップの重要性が増す** とされています。つまり、少数精鋭への投資集中は、コスト削減と競争力強化の両方を実現する戦略になり得ます。

技術顧問として率直に言えば、この「再配置」の議論を経営と建設的に進められるかどうかが、今のCTOにとって最も重要なスキルの一つだと感じています。技術の話だけでなく、人と組織の話を経営の言葉で語れるかどうか。ここが問われています。

## おわりに — 問いを変えることから始める

「エンジニアは何人必要か？」

この問いに対して、もう人数では答えないでください。代わりに、こう問い直してみてください。

**「この組織には、どのケイパビリティが、どのレバレッジ比率で必要か？」**

AIは技術組織の「できること」の上限を引き上げました。しかし同時に、「何をすべきか」を判断する力の重要性も引き上げました。その判断をするのは、今のところ人間です。

人数の最適化ではなく、ケイパビリティの最適配置。これが、AI時代の技術組織サイジングの基本原則です。

:::message
まずは来週、自チームの1週間の業務をHuman-Core / AI-Replaceable / AI-Amplifiableに分類してみてください。おそらく、思っていた以上にAI-Amplifiableな業務が多いことに気づくはずです。
:::
