## はじめに — 「作るか買うか」はもう古い問いかもしれない

技術責任者であれば、「この機能は自社で作るべきか、SaaSを買うべきか」という問いに何度も向き合ってきたはずです。Build vs Buy。ソフトウェア開発における古典的な意思決定フレームワークであり、今なお多くの経営会議で繰り返されている問いでもあります。

しかし、この二択はもう機能しなくなりつつあります。

AIコード生成ツールやエージェントフレームワークの進化により、「ゼロから構築する」でも「既製品を購入する」でもない、**第三の調達モード**が現実の選択肢として浮上しています。

:::message
ある調査では **71%の技術チーム** がオフザシェルフのソリューションを選択しているという報告があります。一方、Deloitteの別の調査では **85%の企業** がAIエージェントを自社ニーズに合わせてカスタマイズすることを前提としています。既製品を「買う」はずが、結局カスタマイズで「作る」に近づいていく——この境界の曖昧さが、従来のフレームワークの限界を浮き彫りにしています。
:::

本記事では、AI前提の時代に技術調達の判断をどう再設計するか、「Build / Buy / Generate」の3層モデルとして提案します。

## なぜ従来のBuild vs Buyが機能しなくなったか

従来のBuild vs Buyには明確な前提がありました。「作る」にはエンジニアの工数と時間がかかり、「買う」にはライセンス費用がかかるが即座に使える。この二軸のトレードオフで判断する、というものです。

この前提を崩している要因が3つあります。

### 1. SaaSの爆発的増加による「選択コスト」の上昇

2020年代を通じてSaaS市場は急拡大し、ほぼあらゆる業務領域に複数の選択肢が存在するようになりました。「買う」という判断をしたとしても、どれを買うかの評価・比較・検証に膨大な時間がかかります。**Buyの意思決定コストそのものが上昇している**のです。

支援先の組織で、タスク管理ツールの選定に3ヶ月かけた末に「結局どれも微妙に要件に合わない」という結論に至ったケースがありました。SaaSが増えるほど選択肢が増え、選択肢が増えるほど「最適解」を探すコストが上がる。この構造的なジレンマは、Buyという選択肢自体の前提を揺るがしています。

### 2. AIによる開発コストの構造的変化

[Claude Code](https://docs.anthropic.com/en/docs/claude-code/overview)や[GitHub Copilot](https://github.com/features/copilot)に代表されるAIコード生成ツールの進化により、特定領域のソフトウェア開発コストが劇的に低下しました。かつてエンジニア数名で数週間かかっていた社内ツールの構築が、数日で完了するケースも珍しくありません。Buildの障壁が下がったことで、従来は「買う一択」だった領域にも「作る」が現実的になっています。

### 3. 「カスタマイズ」と「構築」の境界の曖昧化

SaaSを導入した上でAPIやAIエージェントを使ってカスタマイズする。あるいは、AIエージェントに社内固有のロジックを組み込んだワークフローを生成させる。こうした作業は「買った」のか「作った」のか。既存の二分法では分類できない領域が急速に広がっています。

[CIO.comの記事](https://www.cio.com/article/4097339/your-next-big-ai-decision-isnt-build-vs-buy-its-how-to-combine-the-two.html)でも「次の大きなAI意思決定は、Build vs Buyではない。両者をどう組み合わせるかだ」と指摘されています。問いの立て方自体が変わりつつあるのです。

## 第三の選択肢 — "Generate"という調達モード

従来のBuild vs Buyは「自社で一から構築するか、既製品を購入するか」の二択でした。しかし今、その間に明確な第三の選択肢が生まれています。

### 本記事で提案するのは「Generate」という調達モード

Generateとは、AIエージェントやコード生成ツールを活用し、自社仕様のソフトウェアやワークフローを「組み上げる」アプローチです。

- **Buildとの違い**: ゼロからアーキテクチャを設計するのではなく、AIの生成能力で高速に形にする。設計に数週間かける代わりに、数日で動くものが手に入る
- **Buyとの違い**: 既製品を購入するのではなく、自社固有の要件に最適化されたものを生成する。SaaSの制約に合わせる必要がない

具体的にどういうものか。

- AIコーディングエージェントを使って、社内のデータ連携ツールを**1〜2日で構築**する
- 既存のSaaS APIを組み合わせ、AIエージェントに**ワークフローの自動化スクリプトを生成**させる
- プロダクトのプロトタイプをAI支援で高速に構築し、**仮説検証のサイクルを短縮**する
- 業務固有のダッシュボードやレポートツールをAIで生成する

これらは「外注先がAIに変わった」というだけの話ではありません。**開発の意思決定サイクルそのものが変わる**のです。

個人的な経験でいうと、ある支援先でCRM連携の社内ツールが必要になった際、従来なら「SaaSを探すか、エンジニアに依頼するか」で1〜2週間は検討に費やしていたところを、AIコーディングエージェントで2日でプロトタイプを作り、実際に使ってもらった上で「これをベースに正式採用するか、やはりSaaSを入れるか」を判断しました。動くものを見てから意思決定する——この順番の逆転は、思った以上に判断の質を上げます。

従来は「作るか買うか」を決めてから実行に移るまでに数週間〜数ヶ月かかっていた判断が、「まずAIで生成してみて、使えるか検証してから本格判断する」というプロセスに変わり得ます。Wolters KluwerのCTOが指摘するように、意思決定プロセスの早い段階で「実験」を挟むことが可能になったのです。

つまり、Build vs Buyという「決断」の前に、Generateという「探索」を置ける。これは意思決定の質を根本的に変える可能性があります。

## 3層判断フレームワーク — Build / Buy / Generate

では、具体的にどう判断すればよいのか。以下の3層モデルを提案します。

### Buy層: コモディティ化した機能

自社で持つ必要のない、標準化された機能群です。

| 領域 | 代表的なサービス |
|------|------------------|
| 認証・認可 | [Auth0](https://auth0.com/), [Firebase Auth](https://firebase.google.com/docs/auth) |
| 決済処理 | [Stripe](https://stripe.com/), Square |
| メール配信 | [SendGrid](https://sendgrid.com/), Amazon SES |
| モニタリング | [Datadog](https://www.datadoghq.com/), [Sentry](https://sentry.io/) |
| コミュニケーション | Slack, Teams |

**判断基準**: 業界横断でほぼ同じ要件。差別化の余地がなく、自社で構築する意味が薄い。セキュリティやコンプライアンスの観点からも専門ベンダーに任せた方が合理的な領域。

### Generate層: 自社固有だが、コアではない領域

ここが3層モデルの核心です。自社のビジネスに固有の要件があるためSaaSでは対応しきれないが、競争優位の源泉というわけでもない領域。

- 社内の業務管理ツール・ダッシュボード
- データパイプライン・連携ワークフロー
- 営業・カスタマーサクセス向けの社内支援ツール
- ドキュメント生成・レポート自動化
- プロトタイプ・MVP・仮説検証用のプロダクト

**判断基準**: 自社固有のロジックが含まれるが、それ自体が顧客に提供する価値の中核ではない。**「あると業務が回る」が「これがあるから顧客が選ぶ」ではない機能**。AIの生成能力で十分な品質を実現でき、かつ高速に構築・変更できることに価値がある領域。

### Build層: 競争優位の源泉

自社のプロダクトやサービスの差別化を直接支えるコアロジック。ここは引き続き自社エンジニアが設計・構築するべき領域です。

- プロダクトの核となるアルゴリズム・ロジック
- 顧客体験を直接左右するUI/UX
- 独自のデータモデル・推論パイプライン
- 競合との差別化を生む技術的な仕組み

**判断基準**: 「これがあるから顧客が当社を選ぶ」と言える機能。外部に任せると競争優位が失われるリスクがある。深いドメイン知識と長期的な技術蓄積が必要な領域。

### 判断のためのマトリクス

この3層の分類は、**差別化への寄与度**と**変更頻度**の2つの軸で整理できます。

| | 差別化への寄与度: 低い | 差別化への寄与度: 高い |
|---|---|---|
| **変更頻度: 高い** | Generate | Build |
| **変更頻度: 低い** | Buy | Build |

差別化への寄与度が低く、変更頻度も低い機能は**Buy**が最適です。差別化への寄与度が低いが変更頻度が高い（業務プロセスの変化に追従する必要がある）機能は**Generate層**の候補です。差別化への寄与度が高い機能は、変更頻度に関わらず**Build層**として自社で保持する価値があります。

:::message alert
この分類は固定的なものではありません。事業フェーズや競争環境の変化に応じて、ある機能がBuild層からGenerate層に移行することもあれば、その逆もあり得ます。定期的な再評価が前提です。
:::

## 「Generate層」の運用で問われること

Generate層を実際に運用するには、従来のBuild/Buyにはなかった課題への対応が必要です。

### 生成コードのガバナンス

AIが生成したコードの品質をどう担保するか。これは「レビュー体制」の問題に帰着します。生成されたコードは動くかもしれないが、セキュリティ上の問題はないか、パフォーマンスは十分か、既存システムとの整合性はあるか。人間のエンジニアによるレビューは引き続き不可欠です。

[Deloitteの調査](https://www.deloitte.com/us/en/insights/topics/technology-management/tech-trends/2026/agentic-ai-strategy.html)によれば、エージェントガバナンスの成熟モデルを持つ企業は **わずか21%** です。Generate層を拡大するなら、ガバナンスの整備を先行して進める必要があります。

### 「誰がメンテナンスするか」問題

AIで高速に生成できるということは、属人化のリスクも変質します。

「このコードを書いた人がいない」という従来の属人化から、「このコードを **なぜ** こう書いたか誰も知らない」という新しい形の属人化が生まれます。

対策として考えられるのは、Generate層のコードには生成時のコンテキスト（要件、制約条件、AIへの指示内容）をドキュメントとして残すプラクティスです。コードそのものよりも **「なぜその形になったか」の記録** が重要になります。

実際、自分がAIで社内ツールを生成する際には、AIへの指示内容（プロンプト）と要件の背景をセットでリポジトリに残すようにしています。半年後に修正が必要になったとき、コードを読むより「当時なぜこの設計にしたか」のメモを読む方が圧倒的に速い。これはGenerate層ならではの運用知見だと感じています。

### コスト構造の重心移動

各層のコスト構造は根本的に異なります。

| 層 | 主なコスト | 見落としがちなコスト |
|----|-----------|---------------------|
| **Build** | 開発工数・人件費 | 長期メンテナンス |
| **Buy** | ライセンス費用 | カスタマイズの限界によるワークアラウンド |
| **Generate** | レビュー・検証工数 | 品質管理体制の構築コスト |

「AIが作ったから安い」と見積もると、運用フェーズで想定外のコストが発生します。Generate層のコスト重心は**開発ではなくレビュー・検証・運用**にあることを認識しておく必要があります。

### 組織の意思決定スピードへの影響

Deloitteの調査では、**42%の組織**がエージェント戦略のロードマップを策定中であり、35%はまだ正式な戦略を持っていません。Generate層の活用は技術的な判断だけでなく、組織としての意思決定プロセスの再設計を伴います。

「まず生成して検証する」というサイクルを回すには、**小さな実験を許容する組織文化**と、実験結果を迅速に評価する仕組みが前提になります。

正直なところ、ここが一番難しい部分です。「とりあえずAIで作ってみよう」と言えるカルチャーがある組織と、「まず要件を固めてから」という組織では、Generate層の活用スピードに大きな差が出ます。技術顧問として両方のタイプの組織を見てきましたが、後者が悪いわけではありません。ただ、小さな実験の「失敗コスト」を正しく見積もれていないケースは多い。実はほとんどの場合、数日のAI生成で得られる学びのコストは、数週間の要件定義より遥かに安いのです。

## 経営会議で使える判断チェックリスト

新しいシステムや機能の検討時に、以下の5つの問いを順に確認してみてください。

:::message
**5つの判断フロー**

1. **この機能は、顧客が当社を選ぶ理由に直結するか？**
→ Yes → Build層の候補。自社エンジニアが設計・構築する価値がある。

2. **業界標準のSaaSで十分に対応できるか？**
→ Yes → Buy層の候補。評価・選定に注力し、構築リソースを割かない。

3. **自社固有の要件があるが、頻繁に変更が必要か？**
→ Yes → Generate層の候補。AIで生成し、変更のたびに再生成・調整する。

4. **生成されたものの品質を検証できる体制があるか？**
→ No → Generate層の前にガバナンス体制の整備が先。

5. **失敗した場合のリカバリーコストは許容範囲か？**
→ この問いが、最終的なGo/No-Goの判断基準になる。
:::

### 既存システムの「再分類」

新規の機能だけでなく、既存のシステムも3層で再分類してみる価値があります。

かつてBuild層として自社構築した社内ツールが、今ではGenerate層として再構築した方が保守コストを削減できるかもしれません。逆に、Buy層として導入したSaaSが、自社の要件との乖離が大きくなり、Generate層で代替した方が柔軟に対応できるケースもあるでしょう。

### フェーズ別の推奨アプローチ

| 事業フェーズ | 戦略 | Generate層の活用 |
|-------------|------|------------------|
| **スタートアップ期** | Buy層を最大化し、Build層は差別化の核心に絞る | プロトタイプ・MVP検証 |
| **成長期** | Generate層を戦略的に拡大 | SaaSでは対応しきれない自社固有の業務プロセスをAIで高速構築 |
| **成熟期** | 3層の最適配分を定期的に見直す | Build層の過剰投資、Buy層のSaaS形骸化を再評価 |

## おわりに — 問いの立て方を変える

「作るか買うか」は、長らく技術責任者にとっての重要な問いでした。しかしAI前提の時代に、この二択にこだわり続けることは、判断の質を下げるリスクがあります。

提案したいのは、問いの立て方を変えることです。

> 「この機能は、Build / Buy / Generateのどの層に属するか？」

この問いを起点にすれば、技術的な判断と事業的な判断を同じフレームワークの中で議論できます。エンジニアリングチームと経営層が、共通の言語で意思決定できるようになります。

もちろん、3層モデルは万能ではありません。実際には層の境界は曖昧であり、一つの機能が複数の層にまたがることもあるでしょう。しかし、「作るか買うか」という二択から「どの層に属するか」という三択に変えるだけで、見落としていた選択肢が見えてくることがあります。

技術顧問として多くの組織と向き合う中で感じるのは、意思決定の質は「答えの精度」よりも「問いの適切さ」に左右されるということです。Build vs Buyという古い問いに縛られていると、目の前にある第三の選択肢を見逃してしまう。

自社の技術調達を、3層モデルで一度棚卸ししてみてください。おそらく、いくつかの機能は「実はGenerate層で十分だった」と気づくはずです。そしてその気づきが、エンジニアリングリソースの再配分と、事業成長の加速につながる第一歩になるかもしれません。
