## 導入: AIが書いたコードとAIが作ったUIは、なぜ「同じ顔」に見えるのか

[Claude Code](https://www.anthropic.com/news/claude-code)を使い始めて数週間。最初は「これすごいな」と感動していたのだけど、だんだんと気づいたことがある。**生成されるコードが、どこかで見たような「均質な顔」をしている**。

同じ頃、[v0](https://v0.dev/)や[Bolt](https://bolt.new/)でUIを作ってみると、これもまた「あれ、これ前にも見たな」という既視感。洗練されているし、動作も完璧。でも、どこか**個性がない**。

そんなタイミングで目にしたのが、2026年のデザイントレンド「**Designed Imperfection**（設計された不完全さ）」だった。AI時代の反動として、手書き風のテクスチャや非対称なレイアウト、意図的な「揺らぎ」をデザインに取り入れる動きが加速しているらしい。

:::message
**問いの提示：** AIが「完璧」に近づくほど、なぜ人間は「不完全さ」を求めるのか？
:::

この逆説的な現象を、技術と認知科学の両面から読み解いていきたい。

---

## 第1章: AIコード生成の「特異点」──何が起きているのか

### Claude Codeの90%がClaude Code自身で書かれている事実

2026年1月、Anthropicは[Cowork（Claude Code for general computing）](https://www.anthropic.com/news/introducing-cowork)を公開した。驚くべきことに、**このツールを開発したコードの大半は、Claude Code自身が書いたもの**だった。

4人のエンジニアが10日間で構築した背景には、「Ralph Loop」と呼ばれる自己改善ループがある。これは、**AIが自律的にコードを書き続け、その成果物を次の入力として改善を繰り返す仕組み**だ。名前の由来は『ザ・シンプソンズ』に登場する諦めずに挑戦し続ける少年、Ralph Wiggumから。

このループは数時間連続で動作し続け、**コンテキストが切れることなく、ファイルシステムに保存された成果物を元に一貫した進化を続ける**。人間が介入する必要はほぼない。

### モデルとエージェントの相互改善ループの仕組み

現在、GitHubの公開コミットの**約4%がClaude Codeによって書かれている**。このペースでいくと、2026年末には**全体の20%以上**に達する予測もある([Claude Blog](https://claude.com/blog/eight-trends-defining-how-software-gets-built-in-2026))。

このフィードバックループが特異点に向かっている理由は単純だ：

| フェーズ | 動き |
|----------|------|
| 1. 学習 | LLMが大量のコードを学習（GitHub等） |
| 2. 生成 | AIが新しいコードを書く |
| 3. 再学習 | 生成されたコードが再び学習データに混入 |
| 4. 収束 | 「AIが書きやすい」パターンに最適化される |

実際、Pythonの採用率は前年比で**7ポイント加速**している([Science研究](https://www.science.org/doi/10.1126/science.adz9311))。これは「AI → Python優先 → さらにPythonで学習」という循環が起きているからだ。

### コード品質が均質化する構造的理由

問題は、この自己改善ループが**コードの多様性を奪う**ことだ。

[CodeRabbitの調査](https://www.coderabbit.ai/blog/state-of-ai-vs-human-code-generation-report)によると、AI生成コードは人間が書いたコードと比べて**1.7倍多くの問題を生む**。しかも、リポジトリの慣習に完璧に従わず、命名パターンやアーキテクチャ規範が「ジェネリックなデフォルト」に漂流する傾向がある。

LLMに大量のコードを生成させると、「10人の開発者が会話せずに書いたような」不整合と重複が発生する。これは**学習データの収束**が原因だ。AI同士が書いたコードを学習し合うことで、パターンが固定化されていく。

:::message alert
**構造的な問題：** AIが生成したコードが再び学習データになることで、多様性が失われ、「AIっぽいコード」が量産される。
:::

## 第2章: 2026年UIトレンド「Designed Imperfection」とは何か

### 「設計された不完全さ」の定義と具体例

Canvaの[2026 Design Trends Report](https://www.canva.com/newsroom/news/design-trends-2026/)によると、**「Imperfect by Design」が2026年最大のデザインムーブメント**とされている。

これは、AIによる磨かれた出力への「AI疲れ」に対する直接的な反応だ。具体的には以下のような要素を指す：

- **手描き線**: 定規で引いたようなまっすぐな線ではなく、微妙な揺らぎのある線
- **不規則な構図**: 整然としたグリッドではなく、意図的にずらした配置
- **ブラシストローク**: デジタルの滑らかさではなく、絵の具の質感が残る表現
- **不揃いなタイポグラフィ**: カーニングルールを破る「Typography Rebellion」

### 手書き風テクスチャ、有機的なアニメーション、非対称レイアウト

さらに、**Analog Hybridization（物理メディアのスキャン）** や **Texture Overload（重いグレイン）** も注目されている。Canvaの調査では、「liminal（境界的）」「uncanny（不気味）」といった美学の検索が**前年比220%増**だという。

ある業界関係者の言葉が象徴的だ：

> 「2026年で最も高価なパッケージデザインは、ナプキンにスケッチしたかのように見える──それがまさにポイント」

つまり、**完璧すぎる仕上がりは、逆に信頼を失う**時代になった。

### Apple・Notion・Linearなど実際のプロダクトに見られる「揺らぎ」の兆候

実際のプロダクトでも、この傾向は現れている。

- **Apple**: iOSのアニメーションに「spring（バネ）」効果を多用し、機械的な動きを避ける
- **Notion**: アイコンや装飾に有機的なイラストを採用
- **Linear**: タスク管理ツールでありながら、UIにグラデーションや柔らかいシャドウを残す

どれも、**機能的には完璧だが、視覚的には「人間の手」を感じさせる工夫**がある。

## 第3章: なぜ人間は不完全さに惹かれるのか──認知科学から読み解く

### 不気味の谷仮説のUI版：「完璧すぎるUIへの違和感」

[不気味の谷（Uncanny Valley）](https://ja.wikipedia.org/wiki/%E4%B8%8D%E6%B0%97%E5%91%B3%E3%81%AE%E8%B0%B7%E7%8F%BE%E8%B1%A1)は、1970年にロボット工学者の森政弘が提唱した概念だ。人間に近いロボットほど親近感を持つが、**あるラインを超えると一気に不気味に感じる**という現象を指す。

これはUIデザインにも当てはまる。完璧すぎるインターフェースは、**人間の期待値を超えすぎて、逆に違和感を生む**。

例えば：

- ピクセルパーフェクトな整列
- ミリ秒単位で揃ったアニメーション
- 全く揺らぎのないタイポグラフィ

これらは美しいが、**「人間が作ったものではない」という無意識の違和感**を生む。

### 処理流暢性（Processing Fluency）と「ちょうどいい摩擦」

認知科学の分野では、[処理流暢性（Processing Fluency）](https://www.uxmatters.com/mt/archives/2011/07/how-cognitive-fluency-affects-decision-making.php)という概念がある。これは、**脳が情報を処理しやすいほど、好意的に感じる**という心理現象だ。

ただし、**流暢すぎると退屈になる**。

| 流暢性のレベル | ユーザー体験 |
|--------------|------------|
| 低すぎる | 理解が困難、ストレス |
| 適度 | 快適、信頼感が生まれる |
| 高すぎる | 退屈、記憶に残らない |

デザインにおいては、**「ちょうどいい摩擦」** が重要だ。完全に滑らかすぎると、ユーザーは何も記憶に残さない。逆に、適度な「引っかかり」があることで、記憶に残り、愛着が生まれる。

### 侘び寂び・金継ぎに見る日本文化の「不完全の美学」との接続

日本文化には、**「侘び寂び」** や **「金継ぎ」** といった、不完全さを美として捉える思想がある。

- **侘び寂び**: 完璧でない、未完成のものに美を見出す
- **金継ぎ**: 割れた器を金で修復し、その傷跡を価値として残す

これらは、**「完璧でないことが、むしろ唯一無二の価値を生む」** という哲学だ。

AIが生成する「完璧だが個性のない」アウトプットと対比すると、**人間が手を加えた「不完全さ」が、差別化の源泉**になるという発想に繋がる。

:::message
**ポイント：** 不完全さは欠陥ではなく、「人間らしさ」を伝えるシグナルになる。
:::

## 第4章: AI時代のプロダクト開発で「味」を出す実践テクニック

### コード面：AI生成コードに「チームの個性」を残すCLAUDE.mdの設計思想

AIが書いたコードに「チームらしさ」を残すには、**CLAUDE.md（チーム共通の指示書）** の活用が有効だ。

例えば、Claude Codeに以下のようなルールを設定できる：

```markdown
## コーディングスタイル
- 変数名は必ず日本語コメントを添える
- エラーハンドリングは必ず具体的なメッセージを含める
- 関数の責務は単一に保つ（50行以内を目安）
```

こうすることで、**AIが生成するコードにも、チームの「方言」が残る**。

### UI面：マイクロインタラクションに意図的な「揺らぎ」を入れる手法

UIにおいては、**マイクロインタラクション**に意図的な不完全さを入れるのが効果的だ。

例：

```css
/* 完璧すぎるアニメーション */
.button {
  transition: all 0.3s ease;
}

/* 人間らしさを残す */
.button {
  transition: transform 0.35s cubic-bezier(0.68, -0.55, 0.27, 1.55),
              opacity 0.25s ease;
}
```

このように、**タイミングやイージング関数を微妙にずらす**ことで、「機械的でない」動きが生まれる。

他にも：

- ホバー時に微妙に角度を変える（`rotate(0.5deg)`）
- 影の位置をランダムに少しずらす
- フォントのカーニングを意図的に崩す

### コピーライティング：AI文章の「正しすぎる」問題と崩し方

AI（特にChatGPTやClaude）が生成する文章は、**文法的には完璧だが、個性がない**。

例えば、AIが書くとこうなる：

> 「弊社のサービスは、最先端のAI技術を活用し、お客様の業務効率を最大化します。」

これを人間らしく崩すと：

> 「正直、AIツールは使ってみないと分からない部分が多いです。でも、一度ハマると戻れないくらい便利なので、ぜひ試してみてください。」

**崩し方のコツ：**

- 一人称を使う（「私たちは」ではなく「僕らは」）
- 話し言葉を混ぜる（「〜だと思います」→「〜だと思うんですよね」）
- 失敗談を入れる（「最初はめちゃくちゃハマりました」）

### レビュー観点：「AIっぽくないか？」をチェックリスト化する提案

最後に、プルリクエストやデザインレビューで使える「AIっぽさチェックリスト」を提案したい。

| 観点 | チェック項目 |
|------|------------|
| コード | ・命名が「chatGPT風」になっていないか？<br>・エラーメッセージが汎用的すぎないか？<br>・コメントが自動生成っぽくないか？ |
| UI | ・全ての要素が完璧に整列しすぎていないか？<br>・アニメーションが機械的すぎないか？<br>・色の組み合わせが「AIのデフォルト」になっていないか？ |
| コピー | ・「弊社」「最先端」「最適化」など、堅い言葉ばかりになっていないか？<br>・感情が全く入っていないか？<br>・個人の体験が書かれているか？ |

このチェックリストを使うことで、**「AIが書いたもの」を「チームが作ったもの」に変換する**プロセスが明確になる。

## まとめ: 完璧と不完全の間に、エンジニアの仕事がある

AIの加速と人間の揺り戻しは、**対立ではなく共存**の関係にある。

- **AIが得意なこと**: 大量のコードを高速で生成する、定型的なUIを整える、文法的に正しい文章を書く
- **人間が得意なこと**: コードに「チームの個性」を残す、UIに「揺らぎ」を入れる、文章に「感情」を込める

2026年のプロダクト開発において、最も重要な設計判断は**「何を自動化し、何に人間の手触りを残すか」**だ。

完璧なコードも、完璧なUIも、AIは書ける。でも、**人間らしさは、AIには書けない**。

:::message
**締めの問いかけ：** あなたのプロダクトで「不完全」を許している場所はどこですか？
:::

そこに、あなたのチームの個性が宿っているはずだ。
